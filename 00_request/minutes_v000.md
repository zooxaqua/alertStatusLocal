# 議事録

## 議題
GitHub Copilotを活用したECサイト価格監視ツールのMVP構築方針

## 目的
最小機能を先に完成させ、確実に動作する価格監視の仕組みを短期間で整備する。  
構成は、ローカル運用のしやすさを重視して **Python + Playwright + ローカル定期実行（cron）** を採用する。
あわせて、監視対象と実行状態をブラウザから管理できる画面を用意し、運用者がCLIなしで設定・実行確認できる状態を目指す。

## 決定事項
- 初期構成は Python + Playwright + ローカル定期実行（cron）とする。
- まずは価格取得と通知のコア機能を優先して実装する。
- 監視対象は `config.json` で管理し、商品名・URL・ターゲット価格を定義する。
- 通知設定（Webhook URL/API Key など）は `config.json` に集約し、変更しやすい構造にする。
- 定期実行はローカル環境で自動化し、cron で1時間ごとに実行する。
- テスト実行を手軽にするため、単発実行コマンドを用意する。
- ローカルWebサーバを起動し、ブラウザ管理画面で監視対象（商品名・URL・ターゲット価格）を追加・編集・削除できるようにする。
- ブラウザ管理画面から「テスト実行」と「通知テスト」を実行できるようにする。
- 必要に応じて Amazon など特定ECサイト向けの価格抽出ロジックを追加する。

## 実施方針

### Step 0: 通知疎通確認の最小実装
まずは価格監視ロジックに入る前に、通知経路のみを検証する。

- `--once` オプションは Step 0 では「Slackへ通知を1回送るのみ」の動作とする。
- Step 0 ではスクレイピングは実行しない（通知疎通確認専用）。
- API Key、Webhook URL、チャンネル情報など通知設定は `config.json` の `notification` セクションに格納する。
- `config.json` は通知設定を差し替えしやすいキー構造にする（例: `provider`, `webhook_url`, `api_key`, `channel`）。
- Step 0 の完了条件は「`--once` 実行でSlack通知が1件届くこと」とする。

### Step 1: 価格取得と通知のコアロジック作成
`main.py` を作成し、以下の要件を満たす。

- `playwright` を使用し、ヘッドレスブラウザで動作させる。
- `config.json` から監視対象の「商品名」「URL」「ターゲット価格」のリストを読み込む。
- 各URLにアクセスし、現在価格を取得する。
- 価格取得用のCSSセレクタは後から調整できるよう、変数化しておく。
- 現在価格がターゲット価格以下の場合、Slack Webhookへ通知する。
- サイト構造の違いを考慮し、価格取得処理は関数として共通化する。
- テストしやすいよう、`--once` オプションで単発実行できるようにする。

### Step 2: 設定ファイルと環境構築
`main.py` で使用する補助ファイルを作成する。

- 監視対象リスト用の `config.json` サンプルを作成する。
- 必要なライブラリを記載した `requirements.txt` を作成する。
- ローカル環境変数設定用の `.env.example` を作成する。
- 手軽なテスト用に、`make test-run`（または同等の実行スクリプト）を用意する。
- 画面要件整理用に `00_request/browser_admin_requirements_template.md` を作成し、実装前に入力する。
- `config.json` に通知設定セクション（Webhook URL/API Key 等）を持たせる。

### Step 3: 自動実行設定
ローカルOSのスケジューラにより、定期実行できるようにする。

- cron 設定例を `README.md` に記載する。
- 実行環境はローカル Linux を想定する。
- Python と Playwright の依存関係、およびブラウザのインストール手順を含める。
- Slack通知設定は `config.json` の通知セクションから取得する。
- 毎時実行の cron 例を用意し、JST運用であることが分かる記載にする。
- お試し通知用に、`--test-notify` オプション（通知のみ確認）を実装する。

### Step 4: 特定ECサイト向けの最適化
必要に応じて、対象サイト別の価格取得ロジックを追加する。

- Amazon.co.jp を対象とする場合は、`#corePrice_desktop` 内の価格テキストから数値のみを抽出できるようにする。
- 比較処理に利用できるよう、正規表現を用いた価格パース関数を追加する。

### Step 5: ブラウザ管理画面の追加
ローカル環境で運用者が使える管理画面を追加する。

- ローカルWebサーバ（例: Flask または FastAPI）を起動し、ブラウザでアクセス可能にする。
- 監視対象一覧画面で、商品名・URL・ターゲット価格・最終取得価格・最終実行時刻を表示する。
- 監視対象の追加・編集・削除を画面操作で行えるようにする。
- 「今すぐ監視実行」ボタンで単発実行（`--once` 相当）を呼び出せるようにする。
- 「通知テスト」ボタンでSlack通知の疎通確認（`--test-notify` 相当）を行えるようにする。
- バリデーション（URL形式、価格数値、必須項目）とエラー表示を実装する。
- 初期版はローカル利用を前提とし、認証は任意実装とする。

#### Step 5-1: 画面一覧
- 監視対象一覧画面（`/`）を用意し、登録済み商品の確認と操作を集約する。
- 監視対象編集画面（`/items/new`, `/items/:id/edit`）を用意し、追加・編集を行えるようにする。

#### Step 5-2: 画面要件
- 監視対象一覧には、商品名、URL、ターゲット価格、最終取得価格、最終実行時刻、通知判定結果を表示する。
- 一覧は最終実行時刻の新しい順で表示し、商品名とURLで検索できるようにする。
- 監視対象が0件の場合は、追加導線付きの空状態メッセージを表示する。
- 追加・編集フォームの入力項目は、商品名、URL、ターゲット価格、価格取得セレクタ（任意）とする。
- 保存成功時は一覧へ戻して成功メッセージを表示し、保存失敗時は入力値を保持したままエラー表示する。
- 実行管理として「今すぐ監視実行」「通知テスト」を配置し、実行中・成功・失敗を画面表示する。

#### Step 5-3: データ要件
- 永続化は初期版では `config.json` を利用し、将来DBへ移行可能な構造にする。
- 1レコードは `id`、商品名、URL、ターゲット価格、価格取得セレクタ、最終取得価格、最終実行時刻を持つ。
- `id` はUUIDで採番し、編集時は `id` を不変とする。
- 更新競合を避けるため、保存時にファイルロックまたは排他制御を行う。

#### Step 5-4: バリデーション/エラーハンドリング
- URLは `http://` または `https://` の形式を必須とする。
- ターゲット価格は0以上の数値を必須とし、異常値は保存不可とする。
- 商品名は1〜100文字、URLは1〜500文字を上限目安とする。
- 同一URLの重複登録は禁止し、重複時はユーザ向けに理由を表示する。
- 価格取得失敗、通知失敗、設定保存失敗を想定し、ユーザ向けメッセージとログを分離して出力する。

#### Step 5-5: 非機能/セキュリティ/テスト
- 起動方法は `make run-web`（または同等のコマンド）で統一し、ローカル環境で即時起動可能にする。
- ローカル単一ユーザ利用を前提とし、応答目標は通常操作で2秒以内を目安とする。
- API Key/Webhook URL は `config.json` 管理とし、画面表示やログへの平文出力を禁止する。
- 最低限の受け入れ条件として、追加・編集・削除、今すぐ監視実行、通知テストがブラウザから成功することを確認する。

## Copilotへの依頼方針
- 各ステップごとにプロンプトを分けてCopilotへ入力する。
- まずはMVPの完成を優先し、その後にサイト別最適化へ進む。
- 生成コードは都度確認し、セレクタや通知条件、ローカル定期実行設定を実運用向けに調整する。

## 次のアクション
1. Step 0のプロンプトで `--once` 通知のみ実行を先に実装する。
2. Step 1のプロンプトを使って `main.py` の価格取得ロジックを生成する。
3. Step 2のプロンプトで `config.json` と `requirements.txt` と `.env.example` を作成する。
4. Step 3のプロンプトで cron 設定例とテスト実行コマンドを追加する。
5. 必要に応じて Step 4 で Amazon向けの価格抽出処理を実装する。
6. Step 5の要件詳細に沿って、ブラウザ管理画面を実装する。
